## 一、大语言模型核心原理

### 1\. AI 回复的随机性来源

大模型并非一次性生成完整答案，而是**逐字预测下一个字的概率**：

- 模型会计算每个候选字的出现概率

- 若永远只选概率最高的字，回答会死板固定

- 实际会按概率随机 \&\#34;抽奖\&\#34;，偶尔选中次优字，这就是随机性的来源

### 2\. 温度（Temperature）的控制作用

温度通过调节概率分布的陡峭程度，控制回复的随机性：

- **公式逻辑**：`新概率 ∝ e^\(原对数概率 ÷ 温度\)`

- **低温（0\~0\.3）**：放大高概率字的优势，几乎只选最优解，回答固定、精准、死板，适合做题、写代码

- **中温（0\.6\~0\.8）**：概率差距适中，偶尔跳出最优解，回答自然流畅，是通用场景的常用值（如 0\.7）

- **高温（1\.0\~2\.0）**：拉平所有字的概率，冷门字也能被选中，回答发散、创意性强，适合写小说、编段子

**一句话总结**：温度越小越保守，答案越固定；温度越大越放飞，答案越随机。

### 3\. 大模型的本质

AI 没有思考、没有记忆、没有理解，它本质是一个**概率预测机器**：

- 所有回答底层都是靠训练学到的文字概率，逐字推测生成

- 训练过程中，模型从万亿级文本中学习语言规律、搭配、逻辑，转化为词与词的概率表

- 看起来 \&\#34;聪明\&\#34; 只是因为概率足够准、句子足够连贯，让人类产生了 \&\#34;它在思考\&\#34; 的错觉

## 二、对话历史管理

### 1\. Context 的作用与存储

`context` 是存储完整对话历史的列表，格式为：

```python
[
    {"role": "system", "content": "系统提示词"},
    {"role": "user", "content": "用户消息"},
    {"role": "assistant", "content": "AI回复"}
]
```

- 每一条对话都是独立的字典，包含`role`（角色）和`content`（内容）

- 对话历史是**一问一答两条独立消息**，而非一对键值，这是 API 要求的标准格式

### 2\. Context 的大小限制

- **Python 列表本身**：没有固定上限，理论上可以存储无限多的对话

- **模型上下文窗口**：真正的限制来自大模型，模型一次只能处理有限长度的 token（约等于汉字），常见的有 8k/16k/32k

- 若对话历史超过模型上限，API 会直接报错

### 3\. 为什么要复制 Context

在构建请求时，会先复制一份 Context，而非直接修改原列表，原因是：

- `self\.context` 存储的是**完整、已确认的对话历史**

- 构建请求只是临时操作，若直接修改原列表，一旦 API 调用失败，会导致历史记录被污染

- 复制后的临时列表可以随意修改，不会影响真实的历史记录

- 这是行业通用的安全写法，保证对话历史永远完整正确

### 4\. 复制的性能消耗

复制 Context 的消耗极小，完全可以忽略：

- Python 列表存储的是对象引用，复制列表只是复制地址，速度达到微秒级

- 对话长度受模型限制，最多只有几百条消息，复制成本极低

- 与网络请求的毫秒级耗时相比，复制消耗几乎可以忽略

### 5\. 系统消息的查找

通过以下代码查找系统提示词：

```python
next((m for m in self.context if m["role"] == "system"), None)
```

- 遍历所有对话，筛选出角色为`system`的消息

- `next\(\)` 会取遍历到的**第一条**符合条件的结果，找到后立刻停止遍历

- 若找不到，返回`None`，避免报错

- 作用是判断是否已经设置过系统提示词，用于后续的修改或新增

## 三、Python 代码基础

### 1\. 类型注解：`List\[Dict\[str, str\]\]`

这是 Python 的类型注解，仅用于提示，不影响运行：

- `List`：表示这是一个列表

- `Dict\[str, str\]`：表示列表里的元素是字典，且键和值都是字符串

- `List\[Dict\[str, str\]\]`：表示这是一个**装了很多字典的列表**，也就是对话历史列表

- 删掉这部分注解，代码功能完全不变，只是给 IDE 和开发者看的标记

### 2\. 类与 self 的使用

- 类中的方法，调用自身的其他方法时，**必须加****`self\.`**

- `self` 代表当前类的实例，`self\.方法名\(\)` 表示调用当前对象自己的方法

- 不加`self`，Python 会去全局找函数，找不到就会报错

- 类中的方法通过`self`才能访问到当前对象的属性（比如`self\.context`）

### 3\. `add\_message`方法

这是类中用于添加对话的自定义方法：

```python
def add_message(self, role, content):
    self.context.append({
        "role": role,
        "content": content
    })
```

- 每次调用，都会往`self\.context`里追加一条独立的消息

- 用户输入和 AI 回复需要**分别调用两次**，因为它们是两条独立的消息

- 这是 API 要求的标准格式，保证对话历史的顺序正确

## 四、API 调用全流程

### 1\. 最小请求体

调用对话 API 的请求体，**必填的只有 2 个字段**：

- `model`：模型名称，比如`deepseek\-chat`

- `messages`：对话消息列表，至少要有一条用户消息

最小可用请求体示例：

```python
payload = {
    "model": "deepseek-chat",
    "messages": [{"role": "user", "content": "你好"}]
}
```

其他字段（如`max\_tokens`、`temperature`）都是可选的，用于调整生成效果。

### 2\. HTTP 请求头

请求头是发送给服务器的附加说明，核心包含两个部分：

- `Content\-Type: application/json`：告诉服务器，请求体是 JSON 格式

- `Authorization: Bearer 你的API\_KEY`：Bearer 认证，用于身份验证，`Bearer`是固定前缀，后面跟 API 密钥

标准请求头示例：

```python
headers = {
    "Authorization": "Bearer sk-xxxxxxxxxxxx",
    "Content-Type": "application/json"
}
```

### 3\. POST 请求

调用大模型 API 必须使用 POST 请求：

- POST 请求用于**向服务器提交数据**，把对话消息、模型参数等发给 AI 服务器

- 区别于 GET 请求（仅用于获取数据）

- 完整的 POST 调用示例：

```python
response = requests.post(url, headers=headers, data=json.dumps(payload))
```

### 4\. 响应状态检查

`response\.raise\_for\_status\(\)` 用于自动检查请求是否成功：

- 若请求成功（状态码 200），什么都不做，程序继续运行

- 若请求失败（4xx/5xx 错误，比如密钥错误、无权限），直接抛出异常，立刻停止程序

- 作用是提前暴露错误，避免后续解析 JSON 时出现看不懂的报错

### 5\. 响应结果解析

API 返回的结果中，提取 AI 回复的代码：

```python
result["choices"][0]["message"]["content"]
```

逐层拆解：

- `result`：API 返回的整个大字典

- `choices`：AI 返回的所有候选回答列表，模型可以一次性返回多条回答

- `choices\[0\]`：取第一条回答，也就是**概率最高、模型最推荐**的回答

- `message`：这条回答的消息字典，包含角色和内容

- `content`：最终提取出 AI 真正的回复文本

### 6\. 多候选回答的排序

AI 返回的多条回答，是**按概率从高到低排序**的：

- `choices\[0\]`：概率最高的回答

- `choices\[1\]`：概率第二高的回答

- 以此类推

- 温度参数只影响生成时的随机程度，不改变这个排序规则
